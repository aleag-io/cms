# Access Control & Data Sharing Model

## Overview

This document defines the full access control model for the Mar Thoma Church Management System (CMS) for the Diocese of North America, with particular attention to **parish data sovereignty** — the principle that each parish owns and controls its data — and the **explicit sharing mechanism** that allows parishes to share defined categories of information with the diocese or with other parishes on their own terms.

---

## 1. Core Principles

### 1.1 Parish Data Sovereignty

1. **Default deny at the parish boundary.** All parish operational data — member records, family records, sacramental records, giving records, financial ledger, and internal communications — is private to that parish by default. No other entity (diocese, other parish, or any user outside that parish) may access this data unless an explicit sharing grant exists.

2. **The diocese is a coordinator, not a supervisor.** Diocese-level roles have administrative authority over the structure of the diocese (creating/deactivating parishes, managing diocese-wide programs) but do not automatically inherit the right to read raw parish data. Access to parish data by diocese users requires either a sharing grant from the parish or falls into the narrow category of always-visible structural information (see Section 2).

3. **Parishes are opaque to each other.** Parish A has no visibility into Parish B's data under any circumstance, except through tightly scoped, explicitly granted cross-parish workflows (e.g., member transfer).

4. **Enforcement is at the database layer.** Row-Level Security (RLS) policies in Supabase PostgreSQL enforce these boundaries for every query. Application-layer checks are an additional layer but are not the primary enforcement mechanism.

5. **All access is audited.** Every read or write of sensitive parish data (sacramental records, giving records, personal member data) is recorded in the audit log with actor, timestamp, and data category accessed.

## 2. Diocese Visibility Tiers

Diocese-level users (Diocese Admin, Diocese Staff, Diocese Report Viewer) see parish data in one of three tiers depending on the data category. Parishes do not control Tier 1; they control Tiers 2 and 3.

### Tier 1 — Always Visible (Structural / Administrative)

These are the minimum facts needed for the diocese to administer the network of parishes. No sharing grant is required. All are non-sensitive and describe the parish as an institution, not its people.

| Data | Diocese Admin | Diocese Staff | Diocese Report Viewer |
|------|:---:|:---:|:---:|
| Parish profile (name, address, pastor, contact, website) | ✅ | ✅ | ✅ |
| Parish status (`active` / `merged` / `closed`) | ✅ | ✅ | ✅ |
| Parish user list (names + roles only, no member data) | ✅ | ❌ | ❌ |
| Diocese-wide events and liturgical calendar | ✅ | ✅ | ✅ |
| Diocese-level programs and organizations | ✅ | ✅ | ✅ |
| Own-diocese audit log (diocese-level actions only) | ✅ | ❌ | ❌ |

### Tier 2 — Aggregate / Anonymized Metrics (Always Computed, Never Raw Records)

These metrics are generated by the system as pre-aggregated or anonymized summaries. Diocese users see counts and totals only — no individual member names, addresses, giving amounts, or sacramental details are ever exposed. These are available without a sharing grant because no personal data is disclosed.

| Metric | Scope | Diocese Admin | Diocese Staff | Diocese Report Viewer |
|--------|-------|:---:|:---:|:---:|
| Total member count | Per parish | ✅ | ✅ | ✅ |
| Members by status (active / inactive / deceased / moved) | Per parish | ✅ | ✅ | ✅ |
| Members by gender (counts only) | Per parish | ✅ | ✅ | ✅ |
| Members by age group (counts only) | Per parish | ✅ | ✅ | ✅ |
| Total family count | Per parish | ✅ | ✅ | ✅ |
| Sacrament counts by type and period | Per parish | ✅ | ✅ | ✅ |
| Event count and aggregate attendance | Per parish | ✅ | ✅ | ✅ |
| Program enrollment count | Per parish / program | ✅ | ✅ | ✅ |
| Total giving (sum USD) by period | Per parish | ✅ | ✅ | ✅ |
| Giving by fund category (sum only) | Per parish | ✅ | ✅ | ✅ |

> **Implementation note:** These metrics are produced by materialized views or pre-aggregated summary tables refreshed on a schedule. Diocese users never query the underlying `members`, `donations`, or `families` tables directly. RLS policies on those tables deny access to all diocese-role queries unless a sharing grant exists.

### Tier 3 — Explicitly Shared (Requires a Parish-Issued Sharing Grant)

Raw or detailed parish data (individual records, contact info, giving detail, sacramental records, financial ledger) is only accessible to diocese users when the Parish Admin has issued an explicit **Data Sharing Grant** for the relevant data category. See Section 3 for the full sharing model.

## 3. Parish Data Sharing Model

### 3.1 Data Categories

The following data categories can be shared independently. A sharing grant always applies to exactly one category from this list.

| Category ID | Name | What It Includes |
|-------------|------|-----------------|
| `member_directory` | Member Directory | Full name, contact info, address, status, photo for all active members |
| `member_demographics_detail` | Member Demographics (Detail) | Full demographic data per member (DOB, gender, education level, work notes, skills, interests). Does **not** include private notes — those are never shareable. |
| `family_records` | Family Records | Household data, addresses, preferred contact, member numbers, envelope numbers |
| `sacramental_records` | Sacramental Records | Individual sacramental history (most sensitive category) |
| `giving_detail` | Giving Detail | Per-family or per-member donation amounts, dates, methods |
| `giving_statements` | Giving Statements | Annual IRS-style giving statements per family |
| `program_roster` | Program / Ministry Roster | Member names + enrollment details for a specific program |
| `financial_statements` | Financial Statements | Income statement, balance sheet, fund balances |
| `ledger_detail` | General Ledger Detail | Full journal entry / transaction history (parish ledger only; organization ledgers are not included) |
| `attendance_detail` | Attendance Detail | Per-member attendance records for events and programs |
| `audit_log` | Parish Audit Log | Full audit log of parish-level actions |
| `communications_history` | Communications History | Message content and delivery records |

> **Private notes are never a shareable data category.** The `private_notes` field on member records is accessible only to clergy within the parish and is excluded from all sharing grants, exports, reports, and data transfers.

### 3.2 Grant Mechanics

A **DataSharingGrant** is a record created by a Parish Admin (or Diocese Admin acting on behalf of a request) that authorizes a specific grantee to access a specific data category for a specific parish.

**Grant attributes:**

| Attribute | Description |
|-----------|-------------|
| `parish_id` | The parish whose data is being shared |
| `data_category` | One of the category IDs listed in 3.1 |
| `grantee_type` | `diocese` or `parish` |
| `grantee_id` | UUID of the diocese or the specific grantee parish |
| `grantee_role_filter` | Optional: restrict to a specific role within the grantee (e.g., only `diocese_report_viewer`) |
| `scope` | `all_records` or `summary_only` or `program_scoped` (for program_roster) or `period_scoped` |
| `scope_detail` | JSON: e.g., `{ "program_id": "uuid" }` or `{ "year": 2025 }` |
| `access_type` | `read_only` (default) or `read_write` (reserved for future use) |
| `granted_by_user_id` | Parish Admin who issued the grant |
| `granted_at` | Timestamp |
| `expires_at` | Optional expiry; grants without expiry are permanent until revoked |
| `is_active` | Boolean; set to `false` to revoke without deleting |
| `revoked_at` | Timestamp of revocation (if revoked) |
| `revoked_by_user_id` | User who revoked |
| `notes` | Free text — reason for grant |

**RLS integration:** The RLS policy for each protected table checks for an active `data_sharing_grant` row matching `(parish_id, data_category, grantee_type/grantee_id)` before allowing access. If no grant exists, the query returns zero rows (not an error).

### 3.3 Sharing Request Workflow

Diocese users may not create grants themselves for parish data. The workflow is:

```
Diocese Admin / Diocese Staff
  → creates a DataSharingRequest (specifies parish, data_category, reason, requested_by)
  → Parish Admin is notified (in-app + email)
  → Parish Admin reviews the request
    → Approves: system creates DataSharingGrant; diocese user can now access that category
    → Rejects: request is closed; no access; Diocese Admin is notified of the rejection
    → Ignores: request auto-expires after 14 days
  → Parish Admin can revoke a grant at any time; revocation is immediate
```

Parish Admins can also proactively create grants without a request (e.g., as part of annual reporting cooperation).

All requests, approvals, rejections, and revocations are recorded in the audit log.

### 3.4 Revocation

- Parish Admin can revoke any active grant at any time through the Parish Settings → Data Sharing screen.
- Revocation takes effect immediately at the RLS layer; no grace period.
- Upon revocation, any cached query results (Vercel Data Cache or ISR) for that parish + category are invalidated.
- Diocese users who lose access via revocation are shown a "This data is no longer available" message; they are not told why.

### 3.5 Diocese Admin Override (Emergency Access)

In exceptional circumstances (e.g., a parish is abandoned, a legal or compliance audit is required), Diocese Admin may invoke a time-limited **Emergency Access** override:

- Creates a special `emergency_access_grant` record (separate from normal grants)
- Requires a documented reason (mandatory free-text field)
- Limited to 7 days; cannot be extended; can be revoked early
- Generates an automated notification to the Parish Admin and the system audit log
- Emergency Access grants are visible to Parish Admin in their Data Sharing screen
- These are highlighted distinctly in audit reports

## 4. Cross-Parish Sharing

Parishes are opaque to each other. The following are the only defined cross-parish data flows.

### 4.1 Member Transfer

When a member moves from Parish A to Parish B:

1. Either parish initiates a Transfer Request — identifies the member and the destination parish.
2. The receiving parish (B) sees only the member's **name, status, and a transfer token** — no detailed record until they accept.
3. Upon acceptance by Parish B's Admin, the system copies the member record to Parish B and marks the original as `transferred`.
4. Sacramental records are copied atomically in the same transaction.
5. Giving history remains with the originating parish (Parish A); only a summary (total giving year-to-date) is transferred.
6. The transfer is logged in both parishes' audit logs.

> **Privacy note:** Parish B cannot "search" for a member at Parish A. Transfer must be initiated; Parish B cannot browse Parish A's member directory.

### 4.2 Diocese Program Enrollment

When a parish member enrolls in a diocese-wide program (e.g., a diocesan RCIA cohort):

- The diocese program coordinator sees only: member name, home parish name, enrollment date, and completion status.
- The coordinator does NOT see contact details, giving history, sacramental records, or any other parish data.
- Contact communications for diocese program participants flow through the home parish's communication system unless the member explicitly opts in to direct diocese contact.

### 4.3 Joint Parish Events

If two parishes co-host an event:

- Both Parish Admins must create a **Joint Event** record that links the event across both parishes.
- Each parish's attendees remain private to their own parish.
- Aggregate attendance (total count) is visible to both parishes and the diocese.
- Individual attendee lists are only visible within each parish.

## 5. Role Refinements for the New Model

The following roles replace or refine the definitions in [user-roles.md](user-roles.md).

### 5.1 Diocese Admin

No change to capabilities, but the following clarifications apply under this model:

- Cannot directly read raw parish member, giving, sacramental, or family records unless an active sharing grant exists.
- **Can** issue DataSharingRequests to parishes.
- **Can** invoke Emergency Access (see 3.5).
- Sees all Tier 1 and Tier 2 data for all parishes without restriction.

### 5.2 Diocese Staff

Revised from the original definition. **Diocese Staff no longer has implicit read access to all parish data.**

- Sees Tier 1 (structural) and Tier 2 (aggregate) data for all parishes.
- Sees Tier 3 (detailed) data only for categories where the parish has issued a grant that includes the `diocese_staff` role filter.
- Cannot issue DataSharingRequests (only Diocese Admin can).
- Cannot access any individual member records, giving records, sacramental records, or family records without an explicit grant.

### 5.3 Diocese Report Viewer (New Role)

A read-only diocesan role for staff who need only aggregate statistics and explicitly shared reports.

- Sees Tier 1 and Tier 2 data for all parishes.
- Sees exported or explicitly shared reports that parishes have published to the diocese report repository.
- No access to any raw parish data even if a grant exists (grants targeting this role are limited to `summary_only` or `period_scoped` scope).

### 5.4 Parish Data Sharing Manager (New Role)

An optional parish-level role that a Parish Admin can assign to a trusted staff member to manage the parish's data sharing configuration.

- Can create, modify, and revoke DataSharingGrants on behalf of the parish.
- Can approve or reject DataSharingRequests from the diocese.
- Cannot access data outside their parish.
- All sharing actions they take are attributed to them in the audit log.

### 5.5 Parish Admin Sharing Responsibilities

Parish Admins are the primary custodians of their parish's sharing settings:

- Access Parish Settings → Data Sharing to view all active grants and pending requests.
- Receive in-app and email notifications for new sharing requests.
- Can set a default sharing policy for new requests (manual approval required, or auto-reject).
- Can view a full history of all sharing events for their parish.

## 6. RLS Policy Design

### 6.1 Policy Structure for Protected Tables

For every tenant-scoped table containing personal or financial data, the RLS policies follow this pattern:

```sql
-- Allow parish-scoped users to access their own parish data
CREATE POLICY "parish_own_data" ON members
  FOR ALL USING (
    parish_id = (auth.jwt()->'app_metadata'->>'parish_id')::uuid
    AND (auth.jwt()->'app_metadata'->>'roles') ? 'parish_admin'
       OR (auth.jwt()->'app_metadata'->>'roles') ? 'parish_staff'
       OR (auth.jwt()->'app_metadata'->>'roles') ? 'ministry_leader'  -- scoped further by program_id
  );

-- Allow diocese-level access only when an active sharing grant exists
CREATE POLICY "diocese_shared_data_access" ON members
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM data_sharing_grants g
      WHERE g.parish_id = members.parish_id
        AND g.data_category = 'member_directory'
        AND g.grantee_type = 'diocese'
        AND g.is_active = true
        AND (g.expires_at IS NULL OR g.expires_at > now())
    )
    AND (auth.jwt()->'app_metadata'->>'diocese_id')::uuid IS NOT NULL
    AND (auth.jwt()->'app_metadata'->>'parish_id') IS NULL  -- diocese-level user
  );
```

### 6.2 Aggregate / Tier 2 Views

Tier 2 metrics are computed by database views or materialized views. Diocese users query these views, not underlying tables. The views contain only counts and totals — no personal identifiers.

```sql
-- Example: diocese-visible member aggregate
CREATE VIEW diocese_parish_member_summary AS
SELECT
  parish_id,
  count(*) FILTER (WHERE status = 'active')   AS active_count,
  count(*) FILTER (WHERE status = 'inactive') AS inactive_count,
  count(*) FILTER (WHERE gender = 'male')     AS male_count,
  count(*) FILTER (WHERE gender = 'female')   AS female_count
FROM members
GROUP BY parish_id;
-- Diocese users have SELECT on this view; SELECT on members table is denied without a grant.
```

### 6.3 Self-Service Member Access

Members can only access their own record and their own family record:

```sql
CREATE POLICY "member_own_record" ON members
  FOR SELECT USING (
    user_id = auth.uid()
  );
```

### 6.4 Ministry Leader Scoping

Ministry Leaders are scoped to members enrolled in their assigned program:

```sql
CREATE POLICY "ministry_leader_program_scope" ON members
  FOR SELECT USING (
    parish_id = (auth.jwt()->'app_metadata'->>'parish_id')::uuid
    AND EXISTS (
      SELECT 1 FROM program_enrollments pe
      JOIN programs p ON p.id = pe.program_id
      WHERE pe.member_id = members.id
        AND p.coordinator_member_id = (
          SELECT id FROM members WHERE user_id = auth.uid()
        )
    )
  );
```

## 7. Audit Requirements for Access Control Events

The following events must always be written to the audit log, regardless of other audit settings:

| Event | Logged Fields |
|-------|---------------|
| DataSharingGrant created | actor, parish_id, data_category, grantee_type, grantee_id, scope, expires_at |
| DataSharingGrant revoked | actor, grant_id, parish_id, data_category, revocation_reason |
| DataSharingRequest created | actor, requesting_diocese, parish_id, data_category, reason |
| DataSharingRequest approved | actor (parish admin), request_id |
| DataSharingRequest rejected | actor (parish admin), request_id |
| Emergency Access invoked | actor (diocese admin), parish_id, data_categories, reason, expiry |
| Emergency Access expired | system, parish_id |
| Diocese-level read of shared Tier 3 data | actor, parish_id, data_category, record_count_accessed |
| Role assigned or revoked | actor, target_user, role, parish_id / diocese_id |

## 8. Data Sharing UI — Parish Perspective

Parish Admins and Data Sharing Managers interact with sharing through **Parish Settings → Data Sharing**:

- **Active Grants panel:** lists all currently active grants with grantee, category, scope, expiry, and a Revoke button.
- **Pending Requests panel:** lists incoming diocese sharing requests awaiting decision, with Approve / Reject actions.
- **Request History:** full log of all past requests, approvals, rejections, and revocations.
- **Emergency Access banner:** prominently displayed if an active emergency access grant exists, with the reason and expiry date shown.
- **Default Policy toggle:** "Require my approval for all requests" (default) vs. "Auto-reject all requests" (high-privacy mode).

---

## 9. Data Sharing UI — Diocese Perspective

Diocese Admins and authorized staff interact with sharing through **Diocese Settings → Data Access**:

- **Sharing Requests panel:** create a new request targeting a specific parish and data category. Reason field is required.
- **Active Access panel:** view all parishes that have granted access, by category. Filter by parish or category.
- **Reports Repository:** view reports that parishes have explicitly published to the diocese (see Section 3 variant: parish-pushed reports).
- **Emergency Access panel:** (Diocese Admin only) create an emergency access grant with mandatory justification.
